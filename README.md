# 🎨 Микросервис генерации изображений (выполнение практикума в рамках курса по ML Engineering)

## 📌 Описание проекта
Данный репозиторий содержит спроектированную архитектуру микросервиса. Сервис выступает в роли "обертки" над генеративными ML-моделями (на примере Google Gemini Image - Nano Banana), позволяя пользователям генерировать изображения по текстовому запросу за внутреннюю валюту (кредиты).

Архитектура спроектирована с упором на безопасность данных, строгую типизацию и возможность легкого масштабирования (добавления новых генеративных моделей в будущем).

---

## 🏛 Базовые сущности (Классы)

В рамках выполнения задания были выделены следующие ключевые сущности, с соблюдением принципа единой ответственности (SRP):

1. **`User`** — Клиент сервиса. Хранит исключительно идентификационные данные профиля.
2. **`BillingAccount`** — Кошелек пользователя. Изолированная сущность для управления балансом (Инкапсуляция финансовой логики).
3. **`MLTask`** — Сущность задачи для ML-модели. Отвечает за жизненный цикл генерации (статусы: CREATED, PROCESSING, COMPLETED), позволяя реализовать асинхронную обработку.
4. **`GeneratedImage`** — Сущность результата. Контейнер с готовым изображением.
5. **`BaseImageGenerator` & `GeminiImageGenerator`** — Иерархия ML-моделей генерации. Задает строгий интерфейс (Полиморфизм).
6. **`TaskHistory`** — Хранилище (лог) истории выполненных задач.

## ⚙️ Реализация принципов ООП (Требования задания)

В коде (`image_service_model.py`) строго соблюдены базовые принципы ООП:

### 1. Инкапсуляция (Защита данных)
Критически важные поля скрыты от прямого доступа извне с помощью модификаторов `private` (двойное подчеркивание `__` в Python).
* **Пример 1:** Баланс пользователя (`self.__balance`) и его ID скрыты. Изменение баланса возможно только через валидацию в методе `deduct_credits()`, что исключает несанкционированную накрутку.
* **Пример 2:** API-ключи внутри моделей (`self.__api_key`) скрыты для предотвращения их утечки в другие части программы.
* **Пример 3:** База истории транзакций (`self.__logs`) защищена от прямого редактирования. Добавить запись можно только через метод `log_generation()`.

### 2. Наследование (Расширение классов)
Создана иерархия ML-моделей для избежания дублирования кода.
* Конкретная модель `GeminiImageGenerator` наследуется от родительского класса `BaseImageGenerator`, получая от него общие поля (`model_name`) и методы (например, логику подключения `connect()`). 

### 3. Полиморфизм (Единый интерфейс)
Для обеспечения полиморфизма использован модуль `abc` (Abstract Base Classes).
* В базовом классе `BaseImageGenerator` задан абстрактный метод `@abstractmethod def generate(...)`. 
* Класс-наследник `GeminiImageGenerator` реализует этот метод под специфику API Google. 
* **Польза:** Основная логика сервиса может вызывать метод `.generate()` у любого объекта модели, не зная заранее, какая именно это нейросеть (Gemini, OpenAI, Midjourney и т.д.).
